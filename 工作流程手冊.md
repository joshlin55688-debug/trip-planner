# 工作流程手冊 — 電腦手機互聯開發系統總覽

這是你（Josh）的個人開發系統的**總目錄**。以後不管在電腦還是手機，想做任何事，先開這份找到對應流程，照著唸給 Claude 聽即可。

---

## 🗺️ 系統長什麼樣子

```
 電腦（Claude Code）──┐
                      ├──→ GitHub repo（唯一的真相來源）
 手機（Claude App）──┘         │ push 到 main
                                ↓ GitHub Actions 自動部署
                        Firebase Hosting（正式網站）
```

核心觀念只有三條：

1. **GitHub repo 是中心** — 程式碼、設定、文件、給 Claude 的交接筆記（CLAUDE.md）、指令檔（.claude/commands/）全部放 repo。任何裝置上的 Claude 只要能開這個 repo，就能無縫接手。
2. **push 到 main = 自動上線** — 不用手動部署，改完推上去，幾十秒後正式網址就是新版。
3. **沒 push 的東西等於不存在** — 每個 Claude 對話都是隔離的沙盒。在某個對話裡建立的檔案/指令，沒推上 GitHub，其他對話（包括手機）就讀不到。⚠️ 這是實際踩過的雷：`/ship-app` 指令曾經只存在某個電腦對話裡，手機端完全找不到。

## 📱 流程 1：日常改東西（最常用）

**在手機或電腦開新對話，起手式照這句唸：**

> 打開 joshlin55688-debug/trip-planner 這個 repo，先讀 CLAUDE.md 了解專案，然後幫我〔你要改的東西，用現象描述就好〕

Claude 會自己：讀交接筆記 → 改程式 → 用瀏覽器實測驗證 → commit → push → 自動部署 → 用白話回報改了什麼。

**你不用做的事**：不用貼 log、不用開開發者工具、不用手動部署。
**要注意的事**：改完等 1～2 分鐘再重新整理正式網址；手機看到舊畫面通常是快取，多刷一次就好。

---

## 🆕 流程 2：把新作品接上這套系統（上架）

適用：電腦上又寫了一個新工具，想讓它「手機也能改、改完自動上線」。

**在電腦的 Claude Code 對話裡說：**

> 照 trip-planner repo 裡的《電腦手機共用編輯workflow.md》，把〔資料夾名稱〕這個專案做成手機可編輯＋自動部署

詳細步驟都在 [`電腦手機共用編輯workflow.md`](./電腦手機共用編輯workflow.md)（電腦端 Claude 寫的完整版），四個階段：

1. 資料夾獨立成 GitHub repo
2. 建部署專用服務帳戶 → 存成 GitHub Secret → 接上 Actions 自動部署
3. 寫 `CLAUDE.md` 交接筆記（架構、驗證方式、**踩過的坑**）
4. 手機開對話指到這個 repo，測試一個小改動確認全通

## 📲 流程 3：把作品變成可安裝的 App（/ship-app）

適用：作品要能「加到主畫面」、有自己的圖示、離線可開。

**在 trip-planner repo 的對話輸入 `/ship-app` 即可**（指令檔在 `.claude/commands/ship-app.md`）。
其他 repo 要用的話，先說一句：「把 trip-planner repo 裡 .claude/commands/ 的指令檔複製到這個 repo」。

它會做的事與踩過的雷，詳見指令檔本身。重點雷區：

- **iOS 不吃 SVG 圖示**，apple-touch-icon 一定要 PNG（180、滿版不透明），否則加到主畫面是空白圖
- **LINE/FB/IG 內建瀏覽器裝不了 PWA**，要偵測並引導「用預設瀏覽器開啟」
- Android 自適應圖示要另做一張**滿版 maskable** PNG
- 改了 `sw.js` 或圖示，**快取版本號（CACHE 常數）要 +1**，否則使用者卡舊版
- 已有部署管道的專案**不要**再架第二個部署（例如已有 Firebase 就別再開 GitHub Pages），網址會分裂

---

## 🔗 流程 4：換一個好看的網址（/rename-site）

適用：預設網址太醜（例如 trip-7ab77.web.app → planngo.web.app）。

**在該 repo 的對話輸入 `/rename-site 新名字`**（指令檔在 `.claude/commands/rename-site.md`）。

重點雷區（都實際處理過）：

- 名字只能小寫英文/數字/連字號，全球唯一
- **瀏覽器資料是「每個網址各自獨立」的** — 換網址等於搬新家，一定要做「資料自動搬家」（舊站轉址頁把 localStorage 打包用 `#m=` 帶到新站匯入），否則使用者開新網址會以為資料全消失
- 舊網址要留轉址頁，而且要把 `?room=` 等參數原樣帶過去，別人手上的舊連結才不會壞
- 舊站要放一支「自殺型 sw.js」（清快取＋解除註冊），否則已安裝舊 App 的裝置會一直卡在舊版
- 建新站名不用進 Firebase 控制台：用一次性 GitHub Actions workflow 拿部署服務帳號跑 `firebase hosting:sites:create` 就行（rename-site 指令檔裡有現成範本）
- 換完提醒自己：**開一次舊網址讓資料搬家 + 重新安裝主畫面 App**

---

## ⚠️ 全系統通用雷區備忘

1. **手機雲端沙盒只連得到 GitHub** — 抓不到任何外部網站（連你自己的正式網址都抓不到）。所以：要給手機 Claude 看的東西一律放 repo；驗證用「從 npm 抓套件到本機模擬」的方式做。
2. **對話之間互相看不到** — 建立的指令、修改的檔案，push 了才算數。
3. **CLAUDE.md 是系統的記憶** — 每修一個 bug，要求 Claude 把「症狀→根因→修法→別再犯」寫進 CLAUDE.md，新對話才不會把修好的東西改壞回去。
4. **一定要求 Claude 實測** — 「改完了」不等於「能用」，要求它用瀏覽器真的跑過流程再 push（trip-planner 的 CLAUDE.md 已寫死這個規則）。
5. **金鑰不進 repo** — 服務帳戶金鑰只放 GitHub Secrets，本機/對話裡的暫存檔要立刻刪。
